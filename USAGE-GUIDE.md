# South China Report — 使用指南与提示词手册

> 版本对应 SKILL V3.2.0 ｜ 面向"调用这个技能生成报告"的使用者，不是改技能源码的开发者。
> 读完这一份，你就知道：这技能能干什么、什么场景用、要喂什么数据、产出长什么样，以及**每次生成报告时该怎么写提示词**，才能做到"维度全、挖得深、抓得住问题、读得下去"。
> 结论先行，无 Emoji，数字口径以 `metrics.json` 为准。

---

## 一句话定位

**把销售/经营数据变成一份"给人读、能决策"的静态自包含 HTML 报告**——像一篇有观点的深度文章，而不是 FineBI 看板、PPT 或 Excel。如果产出物和看板没区别，就算失败。

它的强项是：叙事结构（Governing Thought + 章节 PAC 闭环）、视觉纪律（三角色字体 / 语义色 / 克制动效）、数据可信（单一数据源 + 自动校验）。
它**不做**：高频刷新监控、纯明细工作簿、交互式下钻 BI、PPTX/Excel 产物。

## 第一部分 · 功能范围

### 1.1 能做什么

| 能力 | 说明 |
|:---|:---|
| 数据管线 | `scripts/prep-source.py` 用 DuckDB 统吃 Excel/CSV/Parquet/SQLite/DuckDB/任意 SQL，自动画像字段角色、建议分析骨架，清洗聚合后产出 `metrics.json`（含总量、各维度 YoY、占比、集中度、趋势、量价拆解 PVM） |
| 方向感知统计洞察 | `scripts/stat-insights.py` 读 `metrics.json` 产出 `insights.json` + 问题清单；MK、稳健 Z、PVM、TopN、Pareto、HHI 均先判断适用性，指标好坏按 `direction` 解释，HHI 无业务政策只作描述 |
| 叙事设计 | 提炼统领性核心发现（Governing Thought），拆 2–4 章故事弧，每章走"现象→归因→对策"（PAC）闭环 |
| 确定性报告编译 | 有 `report-spec.json` 时由 `scripts/render-report.mjs` 校验受控组件、真源路径、Evidence 和数字绑定后生成 HTML；相同输入逐字节复现，失败不留下最终文件 |
| 多维分析骨架 | 内置"经营分析覆盖面清单"：总量与量价 / 结构升级 / 渠道结构 / 区域战区 / 时间趋势，默认 4–5 章成型 |
| 专业图表 | 选图引擎强制"业务问题→意图→图表"路由；内置瀑布图、子弹图、赛马条形、增长下滑双榜、战区×月热力表、Small Multiples 等；遵循 IBCS 视觉语法 |
| 双风格档 | 紧凑销售报告风（默认，信息密集扫读）与叙事标准风（显式可选，大留白沉浸） |
| 质量保障 | 4 组 Gate：结构校验 → 静态数字核对 → 离线内联严格复检 → 离线版运行时 DOM/ECharts 真值 + 四视口 DOM/AX/Tab/对比度与截图自看 |
| 离线交付 | 一键把外链内联成单文件 HTML，飞书/内网/断网/截图场景可用 |

### 1.2 不做什么（边界）

- **不做高频运营监控看板**（`operational_monitor`）——那是给人"扫"的，走别的工具。
- **不做纯明细分析工作簿**（`analysis_workbook`）。
- **不产出 PPT/Excel/PDF 源文件**——产物是自包含 HTML（可另行截图或打印为 PDF）。
- **不编造数据**：`metrics.json` 标记 `BLOCKED`（如维度跨年编码变更、占比不合）时**只出修数建议，不出结论**；无目标/预算字段时**禁止编造达成率**，自动降级为同比/结构分析。
- **不做交互式下钻**：产物是静态叙事，不是可筛选的 BI 平台。

### 1.3 拦截力的真实边界（别被"通过"骗了）

成品模式会硬阻断 Emoji、数字排版未覆盖、图表无高度、占位符/待校验状态和未确认审计 PASS；`--strict-offline` 还会硬阻断外部与相对资源。饼图误用、双 Y 轴、弱 Hero 标题等仍是 **P1 非阻断项**：`exit 0` 只表示当前模式无 P0，不等于可交付。

## 第二部分 · 适用场景

### 2.1 按报告类型

| 报告类型 | 适用 | 默认模板 | 典型诉求 |
|:---|:---:|:---|:---|
| 月报 / 旬报 | 是（核心） | `scroll-narrative-skeleton.html` | 讲清"发生了什么、为什么、怎么办" |
| 季报 / 年报 | 是（核心） | `scroll-narrative-skeleton.html` | 战略叙事、对外汇报、沉浸阅读 |
| 专题复盘 | 是（核心） | `scroll-narrative-skeleton.html` | 单一议题深挖归因 |
| 决策简报 | 是（简报变体） | `bento-brief.html` | 一屏看完风险/机会/动作 |
| 审计包 / 口径包 | 是（审计变体） | `audit-pack.html` | 证明数据可信、可复算 |
| 高频运营监控 | 否 | —— | 走看板工具 |
| 纯明细工作簿 | 否 | —— | 走 Excel/BI |

### 2.2 按第一读者（决定信息预算）

| 层级 | 第一读者 | 目标 | 信息预算（叙事标准档） |
|:---|:---|:---|:---|
| L1 决策 | 大区总经理、经营管理层 | 3 分钟判断风险/机会/动作优先级 | 3–5 KPI、3–4 主图、2–3 行动 |
| L2 战术 | 战区经理、品类/渠道负责人 | 找责任范围、原因假设、下一步 | 5–8 行动、6–8 图、关键表 |
| L3 分析 | 分析师、项目 owner | 复算口径、追明细、二次分析 | 明细表、字段口径、审计记录 |

> 一份报告可服务多层，但**第一屏只能服务一个第一读者**，其余下沉后续章节或明细区。

### 2.3 按风格档（未指定即紧凑，调用时烘焙进产物）

- **紧凑销售报告风（默认）**：Hero 收为横幅、KPI 加密、密集图表形态。用于销售旬报快速扫读、打印存档、移动端长图。KPI 上调到 5–8、章节 3–6。用户不写风格时直接使用此档。
- **叙事标准风（显式可选）**：大留白、慢节奏、强 Hero 沉浸。仅在用户明确要求沉浸式高管月报/年报或对外汇报时使用。
> 紧凑 ≠ 简陋：密度靠换密集图表形态（子弹阵列/Small Multiples/Sparkline 表格/热力表），**禁止砍归因图和达成图**。产物不含任何页面内切换控件。

## 第三部分 · 输入数据要求

### 3.1 数据源格式

`prep-source.py` 经 DuckDB 统一加载，支持：`.xlsx` / `.csv` / `.parquet` / `.sqlite` / `.duckdb` / 任意 SQL 查询。数据源可以是文件，也可以是库。

### 3.2 字段要求（越全，能成的分析维度越多）

技能按字段角色和通用度量合同自动映射到"哪些维度能成章"。以下是能力角色，不是必须出现的销售字段：

| 字段角色 | 例子 | 支撑的分析维度 | 缺失后果 |
|:---|:---|:---|:---|
| 时间列 | 日期 / 年月 / 旬 | 趋势波动、YoY/MoM 同环比 | 无法做趋势与同比章节 |
| 主度量 | 金额 / 数量 / 人数 / 时长 / 得分 / 比率 / 库存 / 缺陷率 | 核心结论、分布、变化与排名 | 必需主度量缺失才 BLOCKED |
| 辅助度量 | 权重、数量、样本数、目标值 | 加权平均、PVM、置信度与目标比较 | 只关闭依赖方法 |
| 维度 | 部门 / 队列 / 仓库 / 缺陷类型 / 调查分组等 | 结构、排名、组间差异与异常 | 只关闭该维度章节 |
| 时间列（可选） | 日期 / 年月 / 周期 | 趋势及时间基线比较 | 无时间时切 `snapshot`，不阻断结构分析 |
| 目标/Benchmark（可选） | SLA / 行业基准 / 计划值 | 目标、基准差距 | 只跳过对应比较，不自动降级成同比 |
| 主键 | 单号 / 人员周期 / 日期+组织 | 粒度漂移与重复键校验 | 无天然主键时显式写 `[]` |

### 3.3 Data Contract 必答项（模糊数据先问清，否则口径会错）

调用时**必须**向数据/业务确认这些合同，写进提示词：

1. **度量语义**：每个 `measures[]` 的维度轴/时间轴聚合、单位、存储尺度、好坏方向、可加性、权重覆盖、是否主指标/必需是否明确？比率的分子/分母是什么？
2. **分析范围**：是 `period` 还是无时间 `snapshot`？时间报告要区分日历环比、等长滚动窗、上一完整期、上一期同阶段、同比或自定义基线。
3. **Schema 与粒度**：业务粒度、主键、必需/可选字段、类型和单位是什么？无天然主键也要显式写空数组。
4. **可比性**：若做时间比较，当前/基线截止日和观测完整度是否对齐？关键编码是否跨期变化？
5. **参考与漂移**：目标/Benchmark/区间/组间比较是否已在 `references[]` 绑定度量和单位？是否需要用 `drift_lock` 对比上次已确认结果？
6. **方法政策**：是否真的满足 PVM/MK/Pareto/HHI 条件？HHI/Top5 是否有业务政策阈值？没有就只描述集中度。

最小口径示例（完整字段说明见 `references/semantic-data-contract.md`）：

```json
{
  "roles": {
    "time": "date",
    "measures": [{
      "id": "resolution_time", "field": "minutes", "semantic_type": "duration",
      "aggregation": "weighted_mean", "weight_field": "tickets", "unit": "minute",
      "direction": "lower_is_better", "additivity": "non_additive",
      "primary": true, "required": true
    }],
    "dimensions": ["queue"]
  },
  "analysis_scope": {
    "mode": "period", "period": "2025-04", "data_as_of": "2025-04-30",
    "comparison_as_of": "2025-03-31",
    "comparisons": [{"type": "previous_complete_period"}]
  },
  "schema": {
    "business_grain": "queue daily aggregate", "primary_key": ["date", "queue"],
    "fields": {"minutes": {"required": true, "type": "number", "unit": "minute"}}
  }
}
```

`period` 支持月、旬、季度、H1/H2、全年、`start..end` 或 `{start,end}`。日志/日粒度数据建议用 `caliber.expected_observations` 声明两期应有行数或不重日期数，默认低于 90% BLOCKED。分析范围内主指标空值、完全重复行或重复值影响度默认超过 5% 也会 BLOCKED；可选字段缺失只关闭对应模块。

时间列若含 `Z` 或 `+08:00` 等时区偏移，必须另填 `caliber.timezone`（IANA 名称，如 `Asia/Shanghai`）来定义业务日期；缺失时脚本会 BLOCKED，避免跨日错算。

### 3.4 数据至上纪律

报告里**每个数字从 `metrics.json` 抄，不手敲**。无 pandas/duckdb 环境时降级：让用户预聚合，但仍须手工核对占比与达成分母，并标注"未经管线校验"。

## 第四部分 · 输出格式说明

### 4.1 产物形态

- **静态自包含 HTML**：可直接浏览器打开、截图、打印为 PDF。
- **离线单文件**（`report.offline.html`）：外链已内联，飞书/内网/断网可用。
- **截图集**：`desktop.png`(1440)、`desktop-1360.png`、`mobile.png`(430)、`mobile-390.png` 四视口整页图 + 各 `data-snap` 分区 PNG。
- **报告真源契约**：HTML 内必须包含 `#south-china-report-meta` JSON，记录 schema/generator、两个数据截止日、完整性、源文件 SHA-256、报告模式、关键指标、`metrics_sha256` 与 `insights_sha256`。同时必须包含 `#south-china-report-evidence-contract`，核心结论用 `data-evidence-id` 绑定到 `metrics|insights` 路径；无证据原因必须显式标为 hypothesis。详见 `references/evidence-contract.md`。

### 4.2 报告结构（Scroll Narrative 默认布局）

```
Hero（60–100vh）      核心发现一句话 + 核心数字动画
KPI Strip（横向一行）  3–5 关键指标 + 同比箭头
Chapter 1..N（≤4）    Action Title + PAC 叙事 + 图表（上方必有结论标题）+ 洞察卡
Data Detail          高密度明细表（不在叙事主线）
Closing（深色）        编号行动项：对象 + 动作 + 期限 + 验证指标
```

### 4.3 交付前必过的 4 组 Gate

有符合 `references/report-spec-contract.md` 的规格时，优先使用完整构建命令：

```bash
node scripts/build-report.mjs \
  --metrics metrics.json --insights insights.json \
  --spec report-spec.json --out-dir 目标/report-build
```

命令只在 staging 中工作，顺序执行 Renderer 和以下全部 Gate；全通过后才原子发布 `report.html`、`report.offline.html`、`shots/`、`logs/` 与 `build-summary.json`。失败时最终目录不发布，诊断目录会保留已完成产物、逐步状态和原始日志。`--skip-snapshot` 仅供开发，返回退出码 `3 / UNVERIFIED`，不能标为成品。

没有规格或使用人工 HTML 兼容流程时，继续逐段执行：

```bash
# ① 结构校验（exit 0=当前模式无 P0，P1 仍需处理）
node scripts/validate-report.mjs 目标/report.html
# ② 数字一致性 + 100% 可见数字覆盖（业务数字=data-metric；非业务数字=data-number-exempt）
node scripts/verify-numbers.mjs 目标/report.html metrics.json --insights insights.json
# ③ 离线内联 + 严格复检（既有输出默认不覆盖，确认替换时加 --force）
node scripts/make-offline.mjs 目标/report.html && node scripts/validate-report.mjs 目标/report.offline.html --strict-offline
# ④a 浏览器渲染后复核最终 DOM 与全部 ECharts series.data
node scripts/verify-runtime.mjs 目标/report.offline.html metrics.json
# ④b 对严格离线版截图（四视口+分区，拦截出站网络/横滚/DOM/AX/Tab/对比度/页面错误，仍须逐张看）
node scripts/snapshot.mjs 目标/report.offline.html 目标/shots/
```

> **没看过截图 = 没完成**。无 Playwright/Chromium 时运行时数字与截图脚本会报“未验证”，此时必须说明原因。ECharts 新报告使用 V2 合同：标量可用 `metrics` 简写，坐标、树和 custom 嵌套数据须用 JSON Pointer 逐叶绑定。自动 AX Tree 面向读屏器结构，但不等于真实 VoiceOver/NVDA 会话；自动 Tab/对比度也不替代认知可用性、图表替代说明与强合规人工验收。完整合同见 `references/runtime-metrics-contract.md`。

## 第五部分 · 提示词手册（本指南核心）

> 你给技能的"调用提示词"，直接决定报告的维度厚度、分析深度、问题锐度和可读性。
> 用法：先用 §5.1 **黄金主提示词**（填空即可），需要强化哪一项就叠加 §5.2–§5.5 对应模块的"加固句"。

### 5.1 黄金主提示词模板（复制填空）

```
用 south-china-report 技能生成一份【报告类型：月报/季报/年报/专题复盘】。

【数据】数据源在 <路径>；先跑 prep-source.py profile→build 出 metrics.json，
报告所有数字从 metrics.json 抄，不要手敲。

【Data Contract】主指标及辅助度量的聚合/单位/方向/可加性/权重：<填写>；
分析范围：<period/snapshot>；时间比较：<同比/环比/上一完整期/自定义/无>；
目标/Benchmark/组间比较：<有/无及绑定指标>；业务粒度/主键/必需与可选字段：<填写>；
比较截止与完整度：<适用时填写>；编码漂移与业务政策阈值：<填写或明确无>。

【风格 + 读者】风格档：<叙事标准风/紧凑销售报告风>；第一读者：<L1 高管/L2 战区经理/L3 分析师>。

【分析维度】只使用输入中真实存在的字段；优先覆盖主指标概览、分布/异常、结构/排名、
适用的比较与贡献、可验证行动。不得注入输入不存在的战区、渠道、产品、客户或销售术语。

【分析深度】每章走 PAC 闭环：现象→归因→对策，不许只摆数字。
每个数据点过"So What"四连问，答不出决策含义的移到 Data Detail。

【问题发现】build 之后跑 stat-insights.py 出 insights.json，
问题清单/趋势显著性/异常月/断崖引擎/贡献分解从它引用（不许徒手扫描）；
趋势措辞以 mann_kendall.significant 为准，非显著不写"趋势确立"；
每个问题配一条可执行对策（对象+动作+期限+验证指标）。

【可读性】Hero 标题写成一句核心发现（不是"X月经营分析"）；
每章标题是 Action Title（对象+变化+方向，≤30 字）；图表上方必有结论标题；
Closing 给编号行动项。

【交付】跑齐 4 组 Gate（validate → verify-numbers → 离线内联严格复检 → verify-runtime + snapshot 逐张看），
把未清零的 P1 逐条说明。
```

### 5.2 加固模块 A · 丰富分析维度

> 目标：不停在 3 章下限，把销售趋势、区域分布、品类对比、客户分层、时间周期波动都铺开。

叠加句（按需选用）：

- "先跑 `prep-source.py profile` 看它建议的分析骨架，**照建议把每个有数据支撑的维度都成一章**，月报默认 4–5 章，不要压成 3 章。"
- "趋势章用**同期对比（YoY）**：x=1–12 月、叠 2–3 年，让季节峰谷和同比差距一眼可读；**不要用连续多年单折线**（会把季节性画成随机噪声）。"
- "区域维加一张**战区×月热力表**，暴露哪个战区哪个月塌陷。"
- "产品维加**增长/下滑双榜**（按绝对增减额排），谁在涨、谁在崩一屏看清。"
- "客户维做**集中度**（Top N 占比 / CR）+ 时间周期波动（旬/周/月节奏），识别大客户依赖风险。"
- "每个核心指标必须带**至少一类对比基准**：时间（同比/环比）、结构（跨战区/渠道）、目标（达成/历史极值）——裸数字不算分析。"
- "数量类指标要搭**质量/效率伴生指标**（如签收额配毛利率/达成率），防止'不计成本冲量'的误导。"

### 5.3 加固模块 B · 加深分析深度

> 目标：不只呈现结果，挖数据背后的原因与关联，给深层业务洞察。

叠加句：

- "每章强制 **PAC 闭环**：Phenomenon（现象）→ Attribution（归因）→ Countermeasure（对策），缺归因或对策的章节不合格。"
- "对每个关键数据点做 **So What 四连问**：说明了什么 → 为什么这样 → 该怎么做 → 对核心发现有何影响；第 4 问答'不影响'的，移到 Data Detail。"
- "归因要落到**可验证的机制**，不要泛泛。例：'增速从 22% 降到 12%，**不是市场变冷，是去年同期基数变高**'——点出真因，不是复述数字。"
- "用**量价瀑布（PVM）**把总额变化拆成'量的贡献 + 价的贡献 + 结构的贡献'，回答'到底靠什么涨/因什么跌'。"
- "做**跨维度关联**：如'某战区下滑'是否由'某渠道萎缩'或'某型号退场'驱动——把区域、渠道、产品三张表交叉，找传导链，而不是各说各的。"
- "Pull Quote 只写**洞察或反转**，禁止复述报告里已有的数字。"

### 5.4 加固模块 C · 强化问题发现

> 目标：自动识别异常数据、下滑趋势、业绩缺口，并给针对性改进建议。

叠加句：

- "build 之后**必跑 `stat-insights.py`**，问题清单直接从 `insights.json` 取；先看 `method_applicability`，只有 `ELIGIBLE/OK` 的 MK、PVM、TopN、Pareto、HHI 才能进入结论。HHI 无显式政策时只能写描述性集中度。"
- "趋势结论看 `mann_kendall`：`significant=true` 才写'趋势性下滑'；`n<8` 或不显著时措辞用'方向上/初步/波动'，不写'趋势确立'。"
- "问题按**主指标影响或已声明风险等级排序**；语义色读取指标方向，低优指标上涨才标不利，`neutral` 不涂风险色。"
- "**每个问题必须配一条可执行对策**：对象 + 动作 + 期限 + 验证指标。例：'南宁战区 3 日内实地走访反馈根因'，不是'加强管理'。"
- "缺口分析要区分**是量的问题还是价的问题、是哪个渠道/型号拖累**——把缺口拆到可归责的最小单元。"
- "无目标数据时，用**同比转负 + 环比恶化 + 排名下滑**替代'达成缺口'来定位问题，不编造目标线。"

### 5.5 加固模块 D · 提升可读性与实用性

> 目标：结构清晰、重点突出，决策者秒懂关键信息。

叠加句：

- "Hero 标题 = **Governing Thought**（一句核心发现），如'逆势突围，结构领跑'；**禁止**'2026 年 X 月经营分析'这种描述标签。"
- "每章标题是 **Action Title**：[对象] + [关键变化] + [方向/归因]，≤30 字。如'传统渠道占比下降 3.2pp，需加速渠道转型'。"
- "**每张图上方必有结论标题**（是结论句不是分类名）：'传统渠道持续下降，O2O 逆势崛起'，不是'各渠道签收额'。"
- "遵循**倒金字塔**：What changed → So what → Why → Now what；首屏左上放最重要结论或最大风险，不放装饰标题。"
- "Closing 用**编号行动项**：每条含对象、动作、期限、验证指标；避免空话 CTA。"
- "**信任机制可见**：数据日期、来源、审计状态、口径边界至少出现一种。"
- "多章（≥4）加**章节锚点导航**，方便决策者跳读。"
- "文案方向词与数据正负号一致（写'增长'时数据必须为正）；数字列等宽 `tabular-nums` 对齐。"
